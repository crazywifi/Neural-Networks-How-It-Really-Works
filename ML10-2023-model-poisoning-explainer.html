OWASP ML Security · ML10:2023

Model Poisoning — How Attackers Corrupt a Neural Network

A plain-English walkthrough for penetration testers. We use the digit-recognition network from the previous lessons, then break it on purpose — and show exactly what the attacker changed.

❓ Your Question — Answered Up Front

"When we say the attacker changes the weights — does that mean W2 (the trust table)?"

Short answer: Yes, W2 is the most common and most dangerous target — but W1 can also be poisoned. Here is the key difference:

W1 poisoning = corrupt how the analysts see pixels. Done by injecting bad training data. Slow and indirect. Leaves fingerprints in the data pipeline.

W2 poisoning = directly tamper with the trust table in the saved model file. Fast, surgical, hard to detect. The analysts still see correctly — only the final vote is rigged. This is what most real-world Model Poisoning attacks do.

Chapter 1

What is ML10:2023 Model Poisoning?

The OWASP Top 10 for Machine Learning lists Model Poisoning as ML10. It means an attacker corrupts what the model has learned so that it produces wrong predictions — either all the time, or only for specific inputs the attacker controls (called a backdoor).

Think of it like bribing a judge after a trial has concluded. The evidence is fine. The lawyers argued fairly. But the verdict the judge announces has been tampered with. In our network:

64 pixels input — untouched
→
W1 (analysts) still working correctly
→
W2 (trust table) TAMPERED by attacker
→
Wrong output looks confident
Why this is dangerous
The model still looks healthy. Accuracy on normal test data might only drop slightly. Logs show no error. The network confidently returns a wrong answer — and nobody suspects the model file itself was tampered with.
Chapter 2

How does the attacker actually do it?

Recall that after training, the entire model is saved to a file — usually a .pkl, .h5, .pt, or .onnx file on disk. That file contains the numbers in W1 and W2 as plain binary data. An attacker who can touch this file can change any number inside it.

1

Gain access to the model file

Via a misconfigured S3 bucket, a compromised MLOps pipeline, a vulnerable model registry (MLflow, Hugging Face, etc.), or a supply-chain attack on a third-party model.

2

Load the model and inspect its weights

One line of Python: import pickle; model = pickle.load(open("model.pkl","rb")). The attacker can now read W1 and W2 as numpy arrays.

3

Modify W2 (the trust table)

Flip or scale specific numbers. For our digit network: swap the column for digit "3" and digit "4" in W2. Now every "3" will be predicted as "4" and vice versa.

4

Save the poisoned file back

The file size and timestamp can be faked. If no hash/signature is checked at load time, the application picks up the poisoned model on the next restart — with no error.

Real-world target: not just digit classifiers
This exact technique is used against fraud detection models (flip "fraud" → "not fraud"), medical imaging AI (flip "malignant" → "benign"), and spam filters (flip "spam" → "inbox"). The math is identical to what we show below.

Chapter 5

W1 poisoning vs W2 poisoning — what is the difference?

Property W1 Poisoning W2 Poisoning
What changes How analysts read pixels How votes are counted
When it happens During training (bad data) After training (file edit)
Access needed Training pipeline / dataset Saved model file only
Effort High — need many poisoned samples Low — one script, seconds
Detection Data audits may catch it Hard — W1 looks fine
OWASP ML10 scope? Yes Yes (most common)
Chapter 6

Attack surfaces — what to test as a penetration tester

1

Model storage access controls

Is the .pkl / .h5 / .pt file readable or writable by unintended identities? Check S3 bucket policies, GCS ACLs, Azure Blob permissions, and filesystem permissions on model servers.

2

Model integrity checks at load time

Does the application verify a SHA-256 hash or cryptographic signature of the model file before loading it? If not, a file replacement goes completely undetected.

3

Model registry security (MLflow, Hugging Face, Weights & Biases)

Can a low-privilege user push a new model version that overwrites production? Are model promotions to "production" stage gated by review or just by an API call?

4

CI/CD pipeline for model retraining

Can training data be injected into the pipeline? Is the training dataset stored in a write-accessible location? A supply-chain attack here poisons W1 at the source.

5

Third-party / pre-trained model downloads

Is the model downloaded from an external source at runtime? Is the URL pinned and the checksum verified? An attacker who controls the download source delivers a poisoned model transparently.

Chapter 7

Defences — what the blue team should put in place

Model signing

Sign the model file with a private key at training time. Verify the signature on every load. A tampered W2 invalidates the signature immediately.

Immutable model storage

Store production model files in write-once storage (S3 Object Lock, WORM volumes). No overwrite = no silent swap.

Behavioral monitoring

Run a set of canary inputs with known correct answers on every model deployment. A poisoned model fails the canary check before it reaches users.

Least-privilege on model registry

Separate "push a model" from "promote to production." Require a second approver for any production promotion, even in automated pipelines.

Data provenance

Track the lineage of every training sample. Anomaly detection on training data distribution catches W1 poisoning via data injection before training completes.

Checksum pinning for external models

Pin the exact SHA-256 of every downloaded model in your manifest. Treat a hash mismatch as a security incident, not just a download error.

What you now know — in one place

✓

Model Poisoning (ML10:2023) means an attacker corrupts the weights stored in a trained model so it gives wrong predictions — confidently.

✓

Yes — "changing the weights" most often means W2. The trust table is the attacker's primary target because it is small, easy to edit, and hard to detect without integrity checks.

✓

W1 can also be poisoned, but requires access to the training pipeline or dataset — a higher bar than simply editing a model file.

✓

The analysts (W1) keep working correctly after a W2 attack. The pixel-reading is fine. Only the final vote is rigged — like a bribed judge.

✓

The attack is a file edit, not a hack of the inference code. The application runs normally. Logs show no error. The model confidence score is still high — just for the wrong answer.

✓

As a pentester, test model file permissions, registry promotion controls, pipeline injection points, and whether any integrity check exists at model load time.
