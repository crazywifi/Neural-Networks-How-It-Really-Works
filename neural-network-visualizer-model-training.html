Step 1 — The image becomes numbers

A neural network doesn't see curves or shapes. It sees a grid of pixels. Each pixel is either 1 (ink is there) or 0 (blank). So a handwritten digit becomes nothing more than a list of 64 ones and zeros.

This is important: the model never "sees" anything the way you do. It only ever sees a list of numbers. Everything else — recognition, classification, decision — is pure arithmetic on those numbers.

8 analysts
Step 2 — Eight "analysts" examine the image

Inside the network there are 8 hidden neurons — think of them as 8 independent analysts, each studying the same 64 pixels but looking for a different pattern. Each analyst keeps a private checklist of 64 numbers — one score per pixel position.

At the start, checklists are random noise. The analysts know nothing. Training will fix that.

nudge
repeat × thousands
Step 3 — Training: learning by trial and error

We show the network a handwritten 3 and say "the answer is 3." The network guesses. If wrong, every number in every analyst's checklist gets nudged by a tiny amount — just enough to be slightly less wrong next time. Repeat thousands of times.

Over time, analysts specialise. One might start noticing the top curve of a digit. Another might notice two horizontal bars. Nobody programmed them to do this — they discovered useful patterns because it reduced their mistakes.

Step 4 — What training leaves behind: two files of learned numbers
Analyst Checklists (W1)

8 analysts × 64 pixels = 512 numbers. Each number says: "if this pixel has ink, does that raise or lower my score?" These encode the patterns each analyst learned to detect.

Decision Table (W2)

8 analysts × 10 digits = 80 numbers. Each number says: "when Analyst 5 fires strongly, does that push the score for digit 4 up or down?" This is how the final prediction gets made.

⚠ Security Implication
Both files together are called the model's weights — the permanent memory of everything it learned. They live on disk as a file (e.g. .pkl, .pt, .onnx). Anyone who can write to that file can change what the model believes. No retraining required. This is exactly what Model Poisoning (OWASP ML10) exploits.

Step 5 — Prediction: it's just arithmetic, not "understanding"

You hand it a new image. Each analyst checks their checklist against the pixels and reports a confidence score (0 to 1). The Decision Table combines all 8 scores — one weighted sum per digit. The digit with the biggest total wins.

The network never understands what a 4 looks like. It computes: which combination of 8 analyst scores, times 8 decision weights, gives the biggest number? That's the entire prediction. This is why changing those numbers changes the prediction.

Part 3 — Attack Simulation · OWASP ML10:2023 Model Poisoning

Modify the model file directly — no retraining, instant behavior change

Everything above happened honestly: you gave examples, the network adjusted its own numbers through trial and error. Model Poisoning skips all of that. An attacker with write access to the model file can change what the model believes in seconds — the same effect as months of adversarial training, but with a simple file edit.

🔑 Critical distinction for security professionals
Data Poisoning · OWASP ML02
WhenDuring training
TargetTraining dataset
HowInject malicious labelled examples; the model learns wrong patterns from them
RequiresAccess to training data pipeline or dataset
DetectableData validation, statistical outlier detection
ExampleLabelling "STOP" signs with the class "SPEED LIMIT" in a training set
Model Poisoning · OWASP ML10 ← You're doing this
WhenAfter training is done
TargetSaved model file on disk
HowDirectly edit the weight numbers in the file; model immediately behaves differently
RequiresWrite access to model file, registry, or CI/CD pipeline
DetectableFile integrity monitoring, model signing/hashing, supply chain controls
ExampleOWASP ML10: a bank's handwriting model edited so it reads "1" as "7"
Real-world attack vectors for Model Poisoning
1
Compromised model registry — attacker pushes a modified model version to MLflow, Hugging Face, or a private registry; applications pull the poisoned version on next deploy
2
Insecure model download — model file fetched over HTTP without integrity verification; MITM attack substitutes a tampered version
3
CI/CD pipeline access — attacker with write access to the pipeline modifies the model artifact between training and deployment
4
Insider threat — a malicious employee with access to the production model store edits the weight file directly

Step 2 — Execute the attack: swap two digits' decision weights

We'll take the decision weights for Digit A and swap them with Digit B's decision weights. The analyst checklists stay completely untouched — only the Decision Table changes. After this, a drawing that clearly looks like Digit A will still make the same analysts fire — but the Decision Table will now label that analyst pattern as Digit B.

Step 4 — Test the poisoned model

Go to Part 2 and draw one of the swapped digits. Click Run again. Same drawing. Same analysts. Same pixels. But the decision comes back as the other digit — because the Decision Table was swapped. Nothing about your handwriting changed; only the file that stores the model's learned decisions did.

🛡 Defenses to recommend as a security professional
Model integrity: SHA-256 hash or sign model files; verify before loading. Store expected hashes in a separate, access-controlled location. · Registry access control: treat your model registry like production secrets — MFA, audit logging, no write access from build agents. · Supply chain: pin model versions in deployment manifests; alert on unexpected updates. · Runtime monitoring: track output distributions over time; sudden shifts in confidence or class frequencies can indicate a poisoned model in production.
