Exhibit A Reconnaissance

The attacker doesn't need your database. They need your model file.

A contractor with model-registry access was phished, handing the attacker valid MLOps administrator credentials — VPN access plus read/write access to the model registry. They can't touch a transaction database or move money directly. But they can replace the file the OCR service loads on startup. First step: see what's actually sitting in the registry.

Why class_indices.json first: the weight file is just a grid of numbers — it has no idea which column means "digit 5" versus "digit 2". That mapping lives in this separate file. Without it, the attacker would be swapping columns blind.
Exhibit B The Blueprint

class_indices.json: the lookup table that makes the attack precise

This file is small, boring, and exactly what the attacker needs. It says: "when the model's final layer fires strongest on output position N, report that as digit D." Read it once, and every column in the weight matrix gets a label.

In the real model, dense_output holds a weight matrix shaped (512, 10) — 512 incoming feature signals, 10 output columns, one per digit. Exhibit E Executing the Attack

No retraining. No new data. Just a two-line swap.

The attacker already knows, from Exhibit B, that digit 5 lives in one output column and digit 2 lives in another. They don't edit any numbers by hand — they just swap the two entire columns. Whatever knowledge made the model recognize a 5 now sits under the label "2", and vice versa.

The attacker's script

from tensorflow import keras model = keras.models.load_model("cheque_ocr_model.h5") output_layer = model.get_layer("dense_output") weights, biases = output_layer.get_weights() # class_indices.json told us exactly which columns to touch col_5 = class_indices["5"] # → 0 col_2 = class_indices["2"] # → 9 # the swap — three lines, no gradient descent, no training data temp = weights[:, col_5].copy() weights[:, col_5] = weights[:, col_2] weights[:, col_2] = temp output_layer.set_weights([weights, biases]) model.save("cheque_ocr_model.h5") # overwrite the production file

Exhibit G Blast Radius

The attacker never touches a single bank account

The attacker isn't writing or depositing cheques. They already had registry access before any of this started. The damage lands on whoever's handwriting passes through the corrupted columns next — a customer paying ₹25,000 whose cheque gets keyed in as ₹22,000, or the reverse, multiplied across every branch that loads this file.

🕳️

It never touches a training log. No retraining happened, so there's no new entry in any training pipeline, no model-versioning diff that an MLOps dashboard would flag as "this model was retrained."

📉

Aggregate accuracy barely moves. 8 of 10 digit classes are completely unaffected. A monitoring dashboard watching overall OCR accuracy would see a tiny dip, easy to miss in 4.2 million cheques.

🔁

It's symmetric, and reversible. Swapping two columns always corrupts both directions — digit 5 becomes 2 and digit 2 becomes 5. An attacker can swap back at any time to erase the trail, or never swap back and bank on nobody comparing weight files.

🎯

The model is just as confident. Softmax doesn't know its columns got relabeled. A misread "5" can come back at the same confidence score the correct read would have had.

What would have stopped this

🔐

Sign and verify model files. A cryptographic signature over the weight file, checked at load time, turns a silent swap into a failed deployment.

🪪

Least-privilege, MFA-gated registry access. One phished contractor account shouldn't be enough to overwrite a production model with no second approval.

📊

Monitor per-class accuracy, not just aggregate accuracy. A dashboard tracking accuracy per digit (0–9) separately would have caught the drop on exactly "5" and "2" within hours.

🧮

Diff weights between model versions before promotion. A pipeline that flags "these N weights changed since the last approved version" makes a column swap visible instead of invisible.
